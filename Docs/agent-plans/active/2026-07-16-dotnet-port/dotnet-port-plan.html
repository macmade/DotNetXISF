Overview

SwiftXISF is a read-only parser for monolithic XISF 1.0 files (PixInsight's native image format), written in Swift 6: roughly 4,979 lines across 30 source files and ~3,573 lines across 31 test files (226 @Test functions using the swift-testing framework). A single XISFFile entry point, opened from a file path or a byte buffer, validates the 16-byte binary preamble and the UTF-8 XML header and exposes the file's images (typed geometry / sample format / color space plus fully decoded opaque pixel bytes), properties, embedded FITS keywords, unit metadata and per-image color/ancillary elements. Pixel and data-block bytes are decoded lazily on access — that is where base64/hex decoding, attachment slicing, decompression, checksum verification and (opt-in) external-file resolution happen.

This plan ports the whole library and its tests to C# / .NET 10. The approach is dependency-ordered and incremental: each milestone ports one layer (or one concern of a large type), adds its tests, and leaves the solution compiling with all tests green in both Debug and Release, so every milestone can be reviewed and committed on its own. The port favors idiomatic C# over a mechanical translation while preserving the public API shape, type structure, names and observable behavior wherever practical, so a SwiftXISF user recognizes the C# API. It deliberately mirrors the completed sibling port (DotNetFITS) — the same conventions, scaffolding and cross-cutting lessons carry over. The scaffolding (DotNetXISF.slnx, the DotNetXISF class library and the xUnit v3 DotNetXISFTests project with InternalsVisibleTo, nullable reference types, overflow checks and warnings-as-errors) is already in place, and the two sample .xisf fixtures are already present at the repository root under Test Files/.

Unlike DotNetFITS — which is pure C# with no third-party dependencies — XISF's data-block codecs force two managed NuGet dependencies (see Q1). Everything else stays inside the .NET base class library.

Questions & answers

The design decisions settled with the maintainer before planning. (Choices carried over unchanged from the DotNetFITS precedent — binary data as ReadOnlyMemory<byte> and the single-exception error model — are recorded in the Notes rather than re-asked here.)

1. SwiftXISF decodes four compression codecs, two of which have no .NET BCL equivalent. Which decompression dependencies should the port take?
Built-in ZLibStream for zlib, plus two pure-managed NuGet packages: K4os.Compression.LZ4 (lz4 / lz4hc) and ZstdSharp.Port (zstd). .NET 10's System.IO.Compression covers only Deflate/GZip/zlib and Brotli — there is no inbox LZ4, and zstd only arrives inbox in .NET 11 — so a net10.0 library must take third-party packages for those two codecs (SwiftXISF itself pulls in the native libzstd). Both chosen packages are pure managed (no native binaries), MIT-licensed, cross-platform and actively maintained, which keeps the Windows CI friction-free. K4os.Compression.LZ4 (~138 M downloads) decodes raw LZ4 blocks directly, and LZ4HC just emits standard LZ4 blocks so the same decoder handles both. ZstdSharp.Port (~240 M downloads) is a C# transpilation currently tracking upstream zstd v1.5.7 — the very version SwiftXISF's libzstd pins — and exposes a one-shot Unwrap into a known output size, which fits XISF's fixed-output decode. The native binding ZstdNet was rejected: it ships per-platform native assets and carries far less adoption and a more intermittent history. The package references are added in the compression milestone (M8), where they are first used, not at bootstrap.
2. SwiftXISF gates SHA-3 checksums by platform availability; .NET's SHA3_256/SHA3_512 throw where the OS lacks SHA-3. How should the port behave?
Mirror SwiftXISF and degrade gracefully. SHA-1 / SHA-256 / SHA-512 are always available. For SHA-3, check SHA3_256.IsSupported / SHA3_512.IsSupported before use; when a file requests a SHA-3 digest but the platform cannot provide it, throw the XISF unsupported error (XISFException.Unsupported(...)) rather than letting PlatformNotSupportedException surface or silently passing verification. This matches the Swift availability-gated behavior; the SHA-3 test adapts to skip/expect-unsupported the way the Swift matchesSHA3 test does.
3. How should the XML header parser be implemented in C#?
Keep the internal XISFElement tree and port XISFXMLParser over System.Xml.XmlReader. Everything downstream parses from the XISFElement tree, and XISFXMLParser is unit-tested in its own right, so preserving that structure keeps the type relationships and the parser tests meaningful. The reader is configured to be namespace-aware, to extract local names, to compare names case-sensitively, and to be safe by construction — DtdProcessing = Prohibit and XmlResolver = null disable DTD and external-entity resolution, matching SwiftXISF's disabling of external entities. LINQ-to-XML was rejected as it would reshape both the internal structure and the parser tests.
4. What C# shape should XISFValue take (a Swift enum with eight cases, including Data and a Date)?
A readonly struct with an XISFValueKind discriminator, static factory methods and nullable typed accessors, consistent with DotNetFITS's FITSValue. It implements IEquatable<XISFValue> with the Swift equality quirk that NaN == NaN (and a matching GetHashCode). The eight cases are boolean, signed integer (long), unsigned integer (ulong), float (double), complex (real/imaginary doubles), string, time-point (DateTimeOffset), and data (ReadOnlyMemory<byte>). A value type is the closest match to Swift's value-type enum and is consistent with the ReadOnlyMemory<byte> binary-data choice; a sealed class hierarchy was rejected as it would allocate per value and diverge from the FITS precedent.

Notes

Conventions

  • Naming: normal .NET conventions (PascalCase members, I-prefixed interfaces), but keep SwiftXISF formatting conventions (Allman braces; spaces inside parentheses, brackets — including attributes, [ Fact ] — and generic angle brackets; spaces inside interpolation holes; aligned =) as the source document requests. One public type per file, organized into subfolders that mirror the SwiftXISF layout.
  • Namespaces & layout: a single flat namespace DotNetXISF; (and namespace DotNetXISFTests;) regardless of folder, so the public API stays recognizable. Folders are organizational only, mirroring SwiftXISF: Core/, Checksum/, Color/, Compression/, DataBlock/, Extensions/, Image/, Model/. Tests mirror the library structure with cross-cutting helpers (TestUtilities, TestError, integration/descriptions) at the test-project root.
  • Visibility: library types are explicitly public by default (per the maintainer's convention); reserve internal/private for genuine implementation details (e.g. XISFElement, XISFXMLParser, XISFDataBlocksFile, and parser-only helpers). The existing InternalsVisibleTo("DotNetXISFTests") makes internal helpers testable. Classes not designed as hierarchies are sealed; leave XISFFile non-sealed to match the non-final Swift class.
  • Members: mutable state mutates through get/set properties rather than fields; an immutable value type's payload (XISFValue) uses private readonly fields. Lazy caches are nullable auto-properties resolved with ??=.
  • Doc comments: C# XML documentation comments (/// with <summary>/<param>/<returns>/<remarks>/<see cref>/<exception>) on every type and member, including internal/private ones. GenerateDocumentationFile is enabled on the library so a missing public doc fails the build (CS1591); it is off on the test project.
  • License header: every file carries the MIT header block (“Copyright (c) 2026, Jean-David Gadina - www.xs-labs.com”).
  • No references to the Swift original in code or doc comments — describe each type on its own terms.
  • Culture independence: all XISF parsing and formatting must be deterministic and culture-invariant — use CultureInfo.InvariantCulture for every numeric parse/format and ordinal string comparisons. This is a correctness requirement of the format. Enforced by a repo-root .editorconfig raising the .NET globalization analyzers (CA1305, CA1304, CA1311, CA1310, CA1309) to build errors. Caveat: the analyzers do not catch culture-sensitive string interpolation of numbers ($"{value}"), so numbers must never be raw-interpolated, and a culture-swap test (run under fr-FR, asserting the exact string) is required wherever formatting is genuinely culture-sensitive (floats, complex, time-points).
  • Enum zero = neutral: order each discriminator/enum so the default/neutral case is the zero value where one exists. Non-exhaustive switches end with default: throw XISFException... so a future case fails loud.
  • Async: the library is synchronous; the Swift tests are async throws only as a swift-testing idiom. The C# port and its tests are synchronous ([ Fact ] methods, no async).
  • Constants: the XISF preamble constants live on XISFFile as public const/static (Signature = "XISF0100", PreambleSize = 16, Namespace = "http://www.pixinsight.com/xisf") to preserve the API. XISFFile.cs is created early holding only these, and completed in the file-layer milestone.

Porting hazards (read before implementing)

  • Little-endian binary fields, explicitly. The preamble header-length (UInt32 at offset 8) and every .xisb index field are little-endian on disk. Read them with System.Buffers.Binary.BinaryPrimitives.ReadUInt32LittleEndian / ReadUInt64LittleEndian — never rely on host endianness or unchecked reinterpretation.
  • Slice-relative byte handling. Swift Data slices keep parent indices; ReadOnlyMemory<byte>.Slice re-bases to 0, so the non-zero-startIndex gotcha disappears — but the Data extension helpers and attachment slicing must be written against offsets/lengths. attachment(position:size:) offsets are absolute from file start; keep blocks as cheap slices of the whole-file buffer (no copy). File.ReadAllBytes is the analog of the memory-mapped whole-file read (the SIGBUS-on-truncation caveat is dropped).
  • zlib is RFC 1950-wrapped. XISF emits a zlib stream (2-byte header + DEFLATE + 4-byte Adler-32). Prefer System.IO.Compression.ZLibStream (decompress mode), which consumes the wrapper directly — simpler than SwiftXISF's manual header validation + DeflateStream raw-inflate. If DeflateStream is used instead, replicate the header-strip / Adler-32-trim exactly. All codecs decode to a known uncompressed size.
  • lz4 / lz4hc = raw blocks. Decode with K4os.Compression.LZ4.LZ4Codec.Decode into a buffer of the declared uncompressed size (raw block, not the LZ4 frame format). lz4hc data decodes through the same path.
  • zstd = one-shot into known size. Use ZstdSharp.Decompressor.Unwrap with the declared uncompressed size. Dispose the Decompressor.
  • Byte-unshuffle index math is load-bearing. Reverse the planar de-interleaving exactly: unshuffled[i] = shuffled[(i % itemSize) * items + (i / itemSize)] over the whole-item region, leaving any trailing partial item unchanged. Sub-blocks concatenate independently-decoded streams, each to its declared size, then the total is length-validated against uncompressedSize.
  • Checksum before decompress, opt-in. Digests are computed over the stored (still-compressed) bytes and verified before decompression, and only when verifyChecksums is set. Algorithm-name matching is case-insensitive across hyphen variants (sha-1/sha1); digest comparison lowercases both sides; hex formatting is x2 invariant. SHA-3 is availability-gated (see Q2).
  • Lazy caches store success and failure. Swift caches a Result<Data, Error> so a failed decode is not retried. Model this in C# with a cached nullable payload plus a captured exception (or Lazy<T> with exception caching), and inherit the documented “not thread-safe” caveat on XISFFile, XISFImage, XISFDataBlock, XISFICCProfile and XISFThumbnail.
  • External locations are opt-in, lazy and local-only. url(...)/path(...) blocks resolve only when allowExternalLocations is set (off in both presets) and only on data/rawBytes access; only local file:// URLs and both path(...) forms are supported — remote URLs are never fetched. Preserve this security posture. The .xisb block-index walk keeps its cycle-detection (visited set).
  • Culture-sensitive parse sites. Parse invariantly and range-check per width: geometry (d1:...:dN:channels), image bounds (low:high), resolution, RGB-working-space triplets, display-function component vectors, complex (re,im), scalar property values, compression sizes, and location index-ids (decimal or 0x hex UInt64). Integer parsing is range-checked to the declared width (reject overflow). Booleans accept 1/true/0/false (words case-insensitive). Time-points are ISO 8601 (with/without fractional seconds) parsed to DateTimeOffset with invariant culture and round-trip styles.
  • Strict vs. lenient threads through every parser. Under allowSpecDeviations, malformed properties/metadata/ancillary children are dropped rather than fatal, unknown enumerated values fall back to their defaults, and reserved-field / version / size / positivity checks relax. The drop-vs-throw decisions live in XISFProperty.ParseList, XISFImage's optional-child parsing, XISFFile's metadata parsing, and each struct's lenient branch — mirror them precisely.
  • Enumerated-value fallback. The generic Swift enumeratedValue<T: RawRepresentable> helper (unknown raw value → default when lenient, throw when strict) becomes a small generic helper over an enum + its declared default.

Test framework mapping

swift-testing → xUnit v3 (on Microsoft.Testing.Platform). See the appendix for the full table. Highlights: @Test → [ Fact ]; #expect(x) → Assert.True/Assert.Equal; try #require(x) → unwrap-or-fail (Assert.NotNull then use); try #require(throws: XISFError.self){ } → Assert.Throws<XISFException>(() => ...); loop-based multi-case @Tests stay a single [ Fact ] with internal loops (no [ Theory ]); @testable import is covered by InternalsVisibleTo. Test resources resolve via [ CallerFilePath ] (the analog of Swift #filePath): walk from TestUtilities.cs up to the repo root and read Test Files/*.xisf, ordinally sorted — the two ~20–26 MB fixtures already sit there. Runtime temp files use Path.GetTempPath() + Guid.NewGuid() in a try/finally with a shared RemoveTemporaryFile helper (kept internal to avoid xUnit's xUnit1013). The Compression suite's verbatim hex-encoded compressed-stream fixtures are copied across unchanged.
